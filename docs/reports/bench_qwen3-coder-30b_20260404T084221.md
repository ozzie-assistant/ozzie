## Observations

### Strengths
- **Tools**: Perfect score. file_write, web_fetch, web_search all work flawlessly
- **Memory**: Perfect score. store_memory, query_memories, implicit recall, markdown SsoT all solid
- **MCP**: Strong. MongoDB MCP init, tool discovery, and untrusted tool gate all work correctly
- **Security**: Sandbox (AST guard + Seatbelt) blocks destructive commands reliably
- **Code generation**: Go write→compile→fix loop works end-to-end (B19)
- **French language**: Consistent French responses throughout

### Failures
- **Autonomy (submit_task/submit_plan)**: The Qwen3-Coder model fails to use submit_task and submit_plan tools correctly. Instead of submitting tasks, it executes commands directly or enters infinite loops. This is the biggest weakness — 0/17 on B09+B10
- **Raw function call emission**: In several tests (B17, B22), the model emitted raw `<function=...>` XML-style tool calls in its text response instead of using the proper tool calling mechanism. This suggests a training mismatch with the tool calling format
- **Timeouts**: Multiple tests (B09, B20, B24, B25) hit timeouts. The model sometimes enters long thinking loops or retry cycles without producing output
- **Schedule execution**: schedule_task can be created (B11, B21) but tasks don't actually execute — the scheduler may not be triggering subtasks
- **Flow control (B29-B31)**: These tests require WebSocket API interaction which isn't accessible via `ozzie ask` CLI

### Regressions vs previous run
- Previous Qwen3-Coder run (2026-03-28): 103/204 → now 138/228 (adjusted for new tests)
- Memory and MCP scores are new wins (not tested in previous run format)
- Autonomy remains the weakest category
